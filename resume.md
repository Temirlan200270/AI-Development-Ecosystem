# Резюме: что сделано и что впереди

Документ фиксирует состояние платформы **Temir** по итогам работы над **Level 2** (безопасное и однозначное исполнение) и обозначает следующий фокус.

---

## Реализовано

### Оркестрация и исполнение (Level 2)

- **Единый gate перед запуском инструмента** — `can_execute_tool_step` (`temir/core/execution_gate.py`): capabilities, preflight (ОС и shell), проверка идемпотентности; результат — `ExecutionDecision` (`allowed`, `reason`, `retryable`, `terminal`, `skipped_idempotent`).
- **Политика retry по кодам**, а не только по полю IR — `temir/core/retry_policy.py` для кодов контракта IR и preflight (что уходит в repair loop, что terminal, например `unknown_action`, `capability_denied`, `graph` и т.д.).
- **Пост-валидация уровня DAG** — `validate_level_completion` (`temir/core/level_validation.py`): все шаги уровня учтены, каждый в закрытом состоянии (`completed`), при включённой идемпотентности нет дубликатов успешного `intent_sha256` на уровне.
- **Идемпотентность шагов** — множество выполненных пар `(task_id, intent_sha256)` на прогон пайплайна; при совпадении intent — skip с событием (`idempotent_skip`); после успешного выполнения — регистрация intent. Сброс множества при новом `execute_full_pipeline`. Управление: `TEMIR_STEP_IDEMPOTENCY` (по умолчанию включено).
- **Платформа** — единый `PlatformContext` и согласованная политика preflight/событий для Windows и Unix (`temir/core/platform_context.py` и связанные модули).

### CLI и наблюдаемость

- Журнал без Web UI: `temir journal runs`, `tail` (в т.ч. `-f`), `cat`.
- Запуск из файла: `--request-file` / `-f` (удобно для `cmd.exe`).
- Каталог артефактов: `-o` и синоним `--output-dir`.

### Репозиторий и артефакты

- **`.gitignore`**: `.env`, виртуальные окружения, кэши Python/pytest/ruff, `.andromeda/`, каталог песочницы **`output/`**, **`execution_results.json`**, **`summary.yaml`**, **`log.txt`**, `*.log`, типичные артефакты сборки (`dist/`, `build/`, `*.egg-info/`).

### Тесты

- `tests/test_level2_closure.py` — политика retry, валидация уровня, idempotent skip в gate.
- Общий прогон: `pytest tests`.

### Документация

- `README.md` обновлён: Level 2 в возможностях, таблица `TEMIR_*`, раздел «Статус реализации и следующие шаги», пояснение про git и артефакты прогона.

### Прочее (из контекста разработки)

- Replay v2: state machine, diff/branch helpers, API и UI для replay (если используется веб-слой).
- Схема событий с полями платформы там, где требуется для `pipeline` / `execution.level` и т.д.

## Нужно реализовать дальше

### Level 3 — минимальный «intelligence core» (по приоритету продукта)

- Строгий **компилятор spec → IR** с явной схемой и понятными ошибками.
- Слой **нормализации execution intent** (единый формат перед хэшем и исполнением).
- Компактный **model router** (выбор модели/цепочки без раздувания архитектуры).

### Спецификация и продукт

- **`spec.yaml`** в репозитории описывает целевую экосистему; закрытие пунктов спека (в т.ч. Flutter, расширенные сценарии) — отдельная дорожная карта, не путать с «закрытием Level 2».

### Эксплуатация и качество

- Секреты только в `.env`, не коммитить `./output` и отчёты (уже в `.gitignore`).
- При `TEMIR_EVENT_SCHEMA_STRICT=1` — следить за совместимостью новых полей в `temir/web/event_schema.py`.
- При кастомном `TEMIR_EVENT_JOURNAL_DIR` — при необходимости добавить путь в `.gitignore`.
- Перед изменениями: `ruff check .`, `pytest`.

## Критерий «готово» для текущего этапа

**Level 2** считается закрытым: одно формальное правило допуска шага, явная политика retry, проверяемое закрытие уровня DAG, идемпотентность по intent на прогон, документированные переменные окружения и игнор артефактов в Git.

Дальнейший сдвиг ценности — **Level 3** и **выполнение дорожной карты по `spec.yaml`**, а не доработка того же слоя безопасности исполнения без новых требований.
