### Ваш следующий шаг: Что реализовывать прямо сейчас?

Исходя из вашего "Мастер-плана", ваш следующий логический шаг — это **начать реализацию "Этапа 2"**.

Я бы порекомендовал начать с **самой важной и сложной задачи**, потому что она повлияет на всю остальную архитектуру:

**Главная задача сейчас:**
> **Перевести `Orchestrator` на асинхронную модель работы с `asyncio`.**

**Почему именно это?**
*   **Фундамент для параллелизма:** Без `asyncio` вы никогда не сможете запустить двух агентов одновременно.
*   **Эффективность:** Пока один агент "думает" (ждет ответа от API Gemini), другой агент может выполнять свою работу. Это кардинально сократит общее время выполнения.

**Как это сделать (упрощенный план):**
1.  Изменить сигнатуры методов в `Orchestrator` (`execute_full_pipeline`, `execute_task`) на `async def`.
2.  Внутри `execute_full_pipeline`, вместо простого цикла `for`, собрать список всех независимых задач.
3.  Использовать `asyncio.gather()` для одновременного запуска всех этих задач.

Это сложная, но невероятно интересная инженерная задача. Как только вы ее решите, ваша система станет на порядок мощнее.

Отлично. Теперь, когда `spec.yaml v5.0 ("Orchestra Blueprint")` — наш главный ориентир, давайте проведем аудит и определим, что нужно реализовать.

### Ваш следующий шаг: Что реализовывать прямо сейчас?

Исходя из вашего нового, более мощного плана, наш следующий шаг становится очевидным. Мы должны начать строить "Оркестр".

**Главная задача сейчас:**
> **Реализовать базовый пайплайн "Архитектор -> Кодер".**

Это означает, что мы научим `Temir` работать не по `spec.yaml`, написанному человеком, а по `spec.yaml`, **сгенерированному другим ИИ**.

**Как это сделать (упрощенный план):**

1.  **Обновить `UniversalAgent`:** Добавить в него новый метод `execute_mercury_task(...)` (или аналогичный) и промпт для роли `BACKEND_CODER`.
2.  **Создать промпт для `SYSTEM_ARCHITECT`:** В папке `temir/prompts/` создать `system_architect.md`, который будет просить ИИ сгенерировать `execution_plan`.
3.  **Переписать `Orchestrator`:**
    *   Его главный метод `execute_full_pipeline` теперь должен сначала вызывать `SYSTEM_ARCHITECT` с общей задачей (например, "Создай CLI для подсчета слов").
    *   Затем он должен взять сгенерированный `execution_plan`.
    *   И только после этого запустить свой старый цикл `for task in execution_plan: ...`, но теперь вызывая `BACKEND_CODER` (который будет использовать Mercury).

Это сложная и невероятно интересная задача. Она превратит ваш `Orchestrator` из простого "исполнителя" в настоящего "прораба", который сначала получает чертежи от архитектора, а потом раздает задания рабочим.
